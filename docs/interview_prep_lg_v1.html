01 Why LangGraph?
The motivation for migrating from a monolithic chain to a graph-based orchestrator
Q
Why did you migrate from chain.py to LangGraph? What was wrong with the chain?
▼

The v3 chain.py was a single monolithic analyze() method — ~80 lines doing query expansion, retrieval, confidence check, prompt building, LLM call, and response formatting in one function. Three concrete problems:

  1. No retry loop: If confidence was low, you got one shot. No corrective RAG — if the first retrieval was bad, the answer was bad. Period.
  2. Untestable: You couldn't test the confidence gate without running the full pipeline (expansion + retrieval + reranking). Each step was tightly coupled.
  3. Rigid branching: The if/elif cascade for confidence routing was hardcoded. Adding a new branch (e.g., a human-in-the-loop step) meant modifying the core function.

LangGraph solves all three: each step is an isolated node, conditional edges handle routing declaratively, and the rewrite → expand → retrieve loop enables corrective RAG.

Interview phrasing: "We migrated to LangGraph to decompose a monolithic chain into testable nodes, enable corrective RAG with a retry loop, and make routing declarative via conditional edges."
Q
Why LangGraph specifically and not LangChain's LCEL, or plain Python orchestration?
▼
OptionProsCons
Plain Python No dependency, full control Must hand-roll state management, retry loops, conditional routing. No visualization, no checkpointing.
LCEL (LangChain) Composable chains, streaming DAG-only — no cycles (no retry loop). Pipe syntax gets unreadable for 12-node graphs with branches.
LangGraph ✅ StateGraph with cycles, conditional edges, Pydantic state, built-in Mermaid viz, checkpointing Extra dependency (langgraph). Learning curve for state schemas.

The killer feature was cycles. LCEL is a DAG — it can't loop back. Our corrective RAG needs rewrite → expand_query which is a cycle. LangGraph's StateGraph supports this natively.

Key point: "We needed cycles for corrective RAG retry, Pydantic state for type safety, and conditional edges for confidence-based routing. LangGraph was the only option that hit all three."
Q
What does LangGraph give you that you didn't have before?
▼
  1. Corrective RAG loop: If the first retrieval has low confidence, we rewrite the query and try again. This wasn't possible in the linear chain.
  2. LLM-driven classification: We removed the analysis_type dropdown from the UI. Now a Classify node asks the LLM to pick the right type. Users just type a query.
  3. Validation node: After generation, a separate Validate node does a groundedness check. If the answer is ungrounded, we can retry.
  4. Mermaid visualization: graph.get_graph().draw_mermaid() gives an auto-generated architecture diagram from the actual code — it's always accurate.
  5. Future checkpointing: LangGraph supports checkpointers for persistence and human-in-the-loop. We haven't enabled it yet (v1), but the architecture is ready.
Q
What stayed the same during the migration? What did you NOT change?
▼

We followed a "wrap, don't rewrite" principle. These components were untouched:

  • StartupRetriever — the hybrid retriever (semantic + BM25 + RRF + cross-encoder reranker)
  • StartupEmbedder — all-MiniLM-L6-v2 model, 384d embeddings
  • ChromaDB — 134K documents already indexed, no re-embedding
  • BM25 index — built at runtime from ChromaDB documents
  • Prompt templates — 6 templates in prompt_templates.py
  • LLM client factory — get_llm_client() for Groq/OpenAI
  • Evaluation pipeline — eval_dataset.py + evaluator.py

The graph nodes are thin wrappers that call into these existing components. The retriever doesn't know it's inside a LangGraph node.

Interview signal: This shows engineering maturity. You don't rewrite working code during a migration — you wrap it. Minimizes risk, preserves test coverage, and keeps the diff small.
02 State Schema Design
Why 3 Pydantic schemas, why multiple inheritance, why not a dict
Q
Why did you use 3 separate Pydantic schemas instead of one big state dictionary?
▼

LangGraph supports schema filtering — you can declare separate Input, Output, and Internal schemas. This gives you three key benefits:

  1. API surface control: The caller only sees InputState (what to send) and OutputState (what comes back). Internal fields like expanded_query, retry_count, reranker_scores never leak out.
  2. Type safety: Pydantic validates at runtime. If a node returns {"confidence": 42} instead of a string, you get a clear error — not a silent bug three nodes later.
  3. Documentation as code: The schema IS the API contract. Any developer can read InputState and know exactly what graph.invoke() accepts.
InputState: { user_query: str } ← caller sends this OutputState: { answer, analysis_type, source_docs, confidence, best_score } ← caller gets this PrivateState: { ...all 11 fields... } ← graph uses this internally
What a dict would look like: graph.invoke({"user_query": "...", "retry_count": 0, "expanded_query": "", ...}) — the caller would need to know about internal fields. Leaky abstraction.
Q
Why does PrivateState use multiple inheritance from both InputState and OutputState?
▼
class PrivateState(InputState, OutputState): expanded_query: str = "" retry_count: int = 0 # ... private fields

This is how LangGraph's schema filtering works. The StateGraph constructor takes three schemas:

graph = StateGraph( PrivateState, # ← full internal state input=InputState, # ← filters input output=OutputState, # ← filters output )

For this to work, LangGraph requires that PrivateState contains all fields from both Input and Output — which multiple inheritance guarantees via Pydantic's MRO (Method Resolution Order).

Alternative: flat composition — manually copy all fields from Input and Output into PrivateState. Works but violates DRY (if you add a field to OutputState, you must remember to add it to PrivateState too). MI keeps them in sync automatically.

Follow-up they might ask: "Isn't MI bad?" → In general, yes for behaviour. But here it's data MI — Pydantic models are just data containers. There's no diamond-of-death risk because these classes have no methods with conflicting behaviour.
Q
Why Pydantic models and not TypedDict or dataclasses?
▼
OptionProsCons
TypedDict Built-in, zero overhead, LangGraph's default example uses this No runtime validation. {"confidence": 42} silently passes. Also: dict access (state["key"]) is stringly-typed — typos are silent.
dataclasses Attribute access, type hints No runtime validation. No serialization helpers. LangGraph support is limited compared to Pydantic.
Pydantic ✅ Runtime validation, attribute access (state.confidence), serialization, schema generation, default values Slightly heavier import. Requires state.field not state["field"].
Key point: "Pydantic gives us runtime type validation, attribute-style access, and LangGraph schema filtering. The minor overhead is worth catching bugs at node boundaries instead of 3 nodes later."

Q
We hit "PrivateState object is not subscriptable" — what happened and how did you fix it?
▼

When we first wrote the nodes, we used dict-style access: state["user_query"] and state.get("confidence", "high"). This works with TypedDict state (which is a regular dict), but Pydantic models are NOT dicts — they're objects.

❌ state["confidence"] → TypeError: not subscriptable ❌ state.get("confidence") → AttributeError: no .get() method ✅ state.confidence → works ✅ getattr(state, "confidence", "high") → works (with default)

We had to search-and-replace across all 8 node files + builder.py. The fix: state["key"] → state.key, and state.get("key", default) → getattr(state, "key", default).

Lesson learned: When LangGraph uses TypedDict, nodes receive a plain dict. When it uses Pydantic, nodes receive a Pydantic model instance. The access pattern is fundamentally different. We should have caught this earlier by running a single smoke test before writing all nodes.
03 Node Architecture
12 nodes, 3 different patterns — factory, plain, DRY helper
Q
Why 12 nodes? Isn't that over-engineered for a RAG pipeline?
▼

It sounds like a lot, but count by responsibility:

  • 4 core pipeline nodes: classify, expand_query, retrieve, confidence_gate — these exist in any RAG with classification + guardrails
  • 5 generate nodes: One per analysis type. These are thin wrappers (3 lines each) around a shared helper. They exist because LangGraph's conditional edges need distinct node names to route to.
  • 3 control flow nodes: validate, rewrite, refuse — these enable corrective RAG

If you remove the 5 generate nodes (which are 3-line wrappers), you have 7 unique logic units. That's not over-engineered — it's each concern having a home.

The "why 5 generate nodes?" answer: LangGraph's add_conditional_edges routes to node names. You can't route to "generate" and then somehow pass analysis_type. You need distinct targets: generate_similar, generate_swot, etc. The DRY helper (_generate_base.py) ensures the actual logic isn't duplicated.
Q
You have 3 different node patterns — factory, plain function, factory+helper. Why not pick one?
▼

Each pattern exists for a reason:

PatternUsed ByWhy
Factory
make_*_node(dep)		 classify, expand, retrieve, validate, rewrite These nodes need external dependencies (LLM client or retriever). The factory captures the dependency in a closure — the resulting function only takes state.
Plain function confidence_gate Pure logic — no LLM, no retriever, no external dependency. Just threshold math on state.best_score. No factory needed.
Factory + DRY Helper 5 generate_* nodes Each needs the LLM client (→ factory) and shares 95% of logic with the others (→ shared run_generation() helper in _generate_base.py).
Interview phrasing: "The pattern follows the dependency. Nodes with external deps get factories. Pure logic gets a plain function. Nodes that share logic get a DRY helper. Using one pattern everywhere would either over-abstract (factory for confidence_gate) or under-abstract (duplicating generate logic 5 times)."
Q
Explain the classify node. Why does the LLM decide the analysis type?
▼

In v3, the user had to pick "similar", "swot", etc. from a dropdown. This had two problems:

  1. UX friction: Users don't know what "ecosystem mapping" means. They just want to type "Tell me about robotics startups."
  2. Misclassification: A user might pick "similar" when their query is really a SWOT. The wrong prompt template = wrong answer format.

Now the Classify node asks the LLM to pick from 5 types based on the query text. Falls back to "similar" if parsing fails.

_CLASSIFY_PROMPT = """ Classify this startup query into exactly one type: - similar: Finding startups like X - swot: Strengths/weaknesses analysis - competitor: Competitive landscape - comparison: Compare X vs Y - ecosystem: Map a startup ecosystem Query: {query} Type:"""
Trade-off: This adds 1 extra LLM call per query. On Groq free tier (30 RPM), this is significant. We accepted this because classification accuracy is better than user guessing, and the call is cheap (max_tokens=20, temperature=0).
Q
What does the validate node actually check?
▼

Three checks, in order:

  1. Documents exist: Are there source_docs? (Sanity check — retrieve should always produce docs, but defensive.)
  2. Score threshold: Is best_score > 0? (If reranker gave all negative scores, the docs are garbage.)
  3. LLM groundedness check: Ask the LLM "Is this answer grounded in these source documents?" This catches hallucination that slipped past the confidence gate.

If any check fails, validation_check = False. The routing edge then decides: retry if retries remain, or accept best-effort.

Q
Why is _generate_base.py a helper and not a node? What's the underscore convention?
▼

_generate_base.py is prefixed with _ to signal it's internal/private — not a graph node, not exported in __init__.py. It's a helper module containing the shared run_generation() function.

It's NOT a node because:

  • It takes analysis_type as a parameter — but LangGraph nodes only take state
  • It would need state to carry which type to generate — adding routing complexity
  • Having separate node names lets conditional edges route directly to generate_swot without an intermediate dispatch
_generate_base.py: run_generation(llm, state, analysis_type="swot") generate_swot.py: make_generate_swot_node(llm) → node(state) → run_generation(llm, state, "swot") generate_similar.py: make_generate_similar_node(llm) → node(state) → run_generation(llm, state, "similar")
Q
Where does the refuse node live? Why is it inline in builder.py?
▼

The refuse node is 2 lines of code — it sets answer to a hardcoded refusal message:

def _make_refuse_node(): def refuse_node(state) -> dict: return {"answer": _REFUSAL_MSG} return refuse_node

Creating a separate file (refuse.py) for 2 lines would be over-engineering. The _ prefix on _make_refuse_node marks it as builder-private. It has no dependencies, no LLM call, no logic — just a constant assignment.

Principle: "Don't create a file for a function. Create a file for a responsibility." Refuse has no responsibility beyond setting a string.
04 Edges & Routing Logic
2 conditional edges — how they route, why the path map, what each branch means
Q
How does route_after_confidence work? Walk me through every path.
▼
def route_after_confidence(state) -> str: confidence = state.confidence # "high" | "low" | "none" retry_count = state.retry_count if confidence in ("high", "low"): return f"generate_{state.analysis_type}" # → "generate_swot", "generate_similar", etc. # confidence == "none" if retry_count < MAX_RETRIES: # MAX_RETRIES = 1 return "rewrite" # try different query return "__end__" # → maps to refuse node

Path table:

ConfidenceRetry CountRoute ToMeaning
highanygenerate_{type}Good retrieval → generate answer
lowanygenerate_{type}Mediocre retrieval → generate with disclaimer
none0rewriteBad retrieval, first attempt → try rewritten query
none≥ 1__end__ → refuseBad retrieval even after retry → give up gracefully
Q
What's the path_map in add_conditional_edges? Why is it needed?
▼
graph.add_conditional_edges( "confidence_gate", # source node route_after_confidence, # routing function { # path map "generate_similar": "generate_similar", "generate_swot": "generate_swot", "generate_competitor":"generate_competitor", "generate_comparison":"generate_comparison", "generate_ecosystem": "generate_ecosystem", "rewrite": "rewrite", END: "refuse", }, )

The path map serves two purposes:

  1. Validation: LangGraph verifies at compile time that every possible return value from route_after_confidence has a mapping. If the function returns "generate_xyz" which isn't in the map, you get an error at build time — not at runtime.
  2. Remapping: END → "refuse" is a remap. The routing function returns "__end__" but we want to go to the refuse node first (to set the refusal message), then refuse → END.
Subtle point: Without the path map, return END would go directly to __end__, skipping the refuse node. The user would get an empty answer. The remap END: "refuse" ensures the refusal message is always set before the graph terminates.
Q
How does route_after_validate decide retry vs end?
▼
def route_after_validate(state) -> str: if state.validation_check: # True = passed return "__end__" # ✅ good answer # Validation failed: if state.confidence == "low": return "__end__" # best-effort (don't retry low confidence) if state.retry_count < MAX_RETRIES: return "rewrite" # retry return "__end__" # exhausted retries

Key design decision: We DON'T retry when confidence was "low". Why? Because low confidence means the retrieval was mediocre — retrying won't find better docs. We prepend the disclaimer and accept the best-effort answer.

Retry only makes sense when confidence was "high" (good docs, but the LLM hallucinated) — a rewritten query might produce docs that constrain the LLM better.

Q
Why only 2 conditional edges? Didn't you have 4 at some point?
▼

In an earlier LG v1 iteration, we had 4 conditional edges:

  1. route_after_classify — branched to type-specific retrieve nodes
  2. route_after_expand — branched to type-specific retrieve nodes (after expansion)
  3. route_after_confidence — branched to a single generate node
  4. route_after_validate — retry or end

When we flipped to Option B (single retrieve, branched generate), edges 1-2 became unnecessary. Classify → expand → retrieve is now a simple linear chain. The only branching happens after confidence (to pick the right generate node) and after validate (retry or end).

Simplification: 4 → 2 conditional edges. Fewer branches = fewer edge cases = easier to reason about.
05 Design Patterns
Closure DI, factory functions, __all__, and why state stays pure
Q
Explain the closure-based dependency injection pattern. Why not use classes?
▼
def build_graph(retriever, llm_client): # Dependencies captured in closures: classify = make_classify_node(llm_client) # llm_client lives in closure retrieve = make_retrieve_node(retriever) # retriever lives in closure graph.add_node("classify", classify) # graph just sees fn(state) → dict

LangGraph nodes must be fn(state) → dict. Dependencies like llm_client can't be passed as arguments. Three ways to inject them:

ApproachProsCons
Global/singleton Simple Untestable. Can't swap LLM client in tests. Hidden coupling.
Class with __call__ OOP pattern, clear Heavier. Boilerplate (__init__, self.llm_client). Serialization issues if LangGraph tries to pickle nodes.
Closure ✅ Lightweight, testable, no serialization issues (closures aren't pickled by LangGraph internals) Less familiar pattern for some devs
Key insight: LangGraph doesn't serialize node functions — it serializes state. So closures work perfectly. The LLM client lives in Python's closure scope, not in the state. State stays pure Pydantic data.
Q
What does __all__ do and why do you use it in __init__.py files?
▼

__all__ is a Python list that controls what from module import * exports. We use it in two ways:

# graph/state/__init__.py from .input import InputState from .output import OutputState from .private import PrivateState __all__ = ["InputState", "OutputState", "PrivateState"]
  1. Explicit API surface: Even without import *, __all__ documents "these are the public symbols." IDEs use it for autocomplete.
  2. Re-export pattern: from bizintel.graph.state import PrivateState works cleanly instead of from bizintel.graph.state.private import PrivateState. The __init__ re-exports.
  3. Prevents leakage: Internal helpers, constants, or imports in __init__.py don't accidentally become part of the public API.
Q
Why does state have to stay "pure data"? Can't nodes store the LLM client in state?
▼

LangGraph state must be serializable — for checkpointing, replay, and debugging. An OpenAI() client object contains:

  • HTTP connection pools (cannot be pickled)
  • API keys (security risk if serialized)
  • Internal thread locks (non-serializable)

If state contained llm_client: OpenAI, LangGraph's checkpointer would fail. Even without a checkpointer, it violates the principle: state = data about the computation, not the computation tools themselves.

Analogy: State is like a patient's chart — it records blood pressure, test results, history. You don't staple the MRI machine to the chart. The MRI machine (LLM client) is a tool that reads the chart and writes results.
06 Option B — Branch Generate, Single Retrieve
The architecture flip — why we went from 5 retrieve nodes to 5 generate nodes
Q
What was the original architecture and what was Option B?
▼

The original design branched on retrieve — 5 type-specific retrieve nodes that all did the same retrieval but were separate for routing. Option B flipped it:

AspectOriginal (Branch Retrieve)Option B (Branch Generate) ✅
Retrieve nodes5 (retrieve_similar, retrieve_swot, ...)1 (single retrieve)
Generate nodes1 (unified generate)5 (generate_similar, generate_swot, ...)
Branch pointAfter classifyAfter confidence_gate
Where type mattersRetrieve (but they all did the same thing!)Generate (where type actually selects the prompt)
Q
Why was branching retrieve a problem?
▼

The 5 retrieve nodes all called the same retriever.retrieve() with the same parameters. They were uniform in behaviour but branched in structure. This violates the principle: "branch where behaviour diverges."

  • 5 identical files: retrieve_similar.py, retrieve_swot.py, etc. — all calling _retrieve_base.py with zero differences.
  • Unnecessary routing: route_after_classify and route_after_expand existed just to direct to the "right" retrieve node — but they all did the same thing.
  • Confusion: "Why does the type matter for retrieval?" It doesn't — retrieval is the same regardless of whether you want SWOT or competitor analysis.
Where type DOES matter: Generation — that's where get_prompt(analysis_type) selects the right template (SWOT format vs competitor table vs ecosystem map). Branching generate is semantically correct.
Q
Were there other options? Why not branch both retrieve and generate?
▼

Three options were on the table:

OptionStructureVerdict
A: Branch both 5 retrieve + 5 generate = 10 branched nodes ❌ Maximum flexibility but massive graph. 10 identical retrieves + 10 conditional edges. Way over-engineered.
B: Branch generate only ✅ 1 retrieve + 5 generate = 6 nodes ✅ Branches at the point of divergence. Linear up to confidence, then fans out.
C: Branch neither 1 retrieve + 1 generate ⚠️ Simplest, but generate would need if analysis_type == "swot": ... inside it. Conditional logic hidden inside a node instead of in edges.
Option C was tempting but violates LangGraph's philosophy: routing decisions belong in edges, not inside nodes. A single generate node with a big switch/case defeats the purpose of a graph orchestrator.
07 Corrective RAG & Retry Loop
How the rewrite loop works, why MAX_RETRIES=1, what prevents infinite loops
Q
Explain the corrective RAG loop. What triggers it?
▼

Two triggers can start a retry:

  1. Confidence = "none" after confidence_gate: The retrieval was so poor (best_score below HARD threshold) that we don't even try to generate. Instead → rewrite the query and try again.
  2. Validation failed after validate: The answer was generated but the groundedness check failed. The LLM hallucinated despite having docs → rewrite and try.

The rewrite loop path:

rewrite → expand_query → retrieve → confidence_gate → generate_* → validate ↓ ↓ (rewrites expanded_query, (checks groundedness, increments retry_count) may loop back again)
Q
Why MAX_RETRIES = 1 and not 2 or 3?
▼

Each retry costs up to 4 LLM calls (expand + generate + validate + possibly classify). On Groq free tier (30 RPM), more retries means more rate limiting. Practically:

  • 0 retries: No corrective ability at all
  • 1 retry ✅: One chance to recover from a bad query expansion. Covers the most common failure mode.
  • 2+ retries: Diminishing returns. If one rewrite didn't help, a second rewrite of a rewrite is unlikely to either. Plus it doubles latency.
Latency math: Base path = 4 LLM calls × ~1s each = 4s. With 1 retry = 7-8s. With 2 retries = 11-12s. Users won't wait 12 seconds. One retry is the sweet spot.
Q
What prevents infinite loops in the graph?
▼

Two safeguards:

  1. retry_count in state: The rewrite node increments retry_count. Both routing edges check retry_count < MAX_RETRIES. Once exhausted → route to END.
  2. LangGraph's built-in recursion limit: Even if our logic had a bug, LangGraph caps the total number of node executions per invocation (default 25). After 25 steps, it raises GraphRecursionError.

Between these two, infinite loops are impossible. The first is our application-level guard; the second is a framework-level safety net.

Q
The rewrite node rewrites the expanded_query, not the user_query. Why?
▼

The user_query is sacred — it's what the user typed. We never modify it. The rewrite node rewrites expanded_query because:

  • The problem is expansion, not the user: If retrieval failed, it's because query expansion produced bad search terms, not because the user asked a bad question.
  • Provenance: We always want to show "You asked: {original query}" in the UI. If we mutated user_query, we'd lose the original.
  • Generate needs the original: The generate prompt uses user_query to produce the answer. It should see "Find startups similar to Stripe", not the rewritten "fintech payment API developer tools".
08 Migration Process
How the codebase evolved from chain.py to the graph — what changed, what broke, what we learned
Q
Walk me through the migration steps. What was the order of changes?
▼
  1. Design phase: Discussed 3 DI approaches (global, class, closure) → chose closure. Discussed 3 state approaches (dict, TypedDict, Pydantic) → chose Pydantic 3-schema.
  2. State schemas: Created graph/state/ — InputState, OutputState, PrivateState with MI.
  3. Nodes: Created all node files. Initial version had 5 branched retrieve nodes (Option A).
  4. Builder + edges: Wired everything in builder.py. Verified it compiled.
  5. Delete old code: Removed rag/chain.py, utils/. Updated all imports in streamlit_app.py, run_eval.py, state.py.
  6. Architecture flip: After reviewing, spotted the asymmetry (retrieve branched but uniform). Flipped to Option B — deleted 5 retrieve_* + _retrieve_base.py, created 5 generate_* + _generate_base.py.
  7. Bug fixes: Fixed Pydantic state access (not subscriptable), added load_dotenv(), fixed rate limiting.
  8. Eval + validation: Ran eval pipeline, diagnosed regression (rate limiting), added max_retries and query delay.
Q
What files did you have to update when replacing chain.py with the graph?
▼

Four consumer files needed updating:

  1. app/state.py — Changed load_chain() to load_graph(). Swapped StartupChain(retriever, llm_client) for build_graph(retriever, llm_client).
  2. app/streamlit_app.py — Changed chain.analyze(query, analysis_type) to graph.invoke({"user_query": query}). Result keys changed from chain's dict format to OutputState fields.
  3. app/components.py — Removed the "Analysis Type" dropdown from the sidebar (classify node now handles this).
  4. evaluation/run_eval.py — Changed chain.analyze() to graph.invoke(). Same result key mapping.
Q
What went wrong during the migration? What bugs did you encounter?
▼

Three significant bugs:

  1. PrivateState not subscriptable: All nodes used state["key"] — works with TypedDict, fails with Pydantic. Had to change 8 files to state.key / getattr(). Root cause: LangGraph docs default to TypedDict examples; we chose Pydantic without adjusting access patterns.
  2. Missing dotenv in notebook: The Jupyter notebook didn't have load_dotenv(), so GROQ_API_KEY wasn't loaded. Graph compiled fine but LLM calls failed with "API key not set". Subtle because the error came from deep inside the Groq client.
  3. Rate limiting causing eval regression: Post-migration eval showed dramatic metric drops. Investigation revealed 429 rate limit errors from Groq (30 RPM limit). The pipeline now makes more LLM calls per query (classify + expand + generate + validate) and the eval fires 30 queries with 7 LLM calls each = 210 calls. Fix: max_retries=3 on the OpenAI client + 2-second delay between eval queries.
Interview power move: Don't hide bugs — narrate them as learning stories. "We discovered the Pydantic access issue early in testing, which reinforced our principle of running smoke tests after each node, not after all 12."
09 Evaluation & Debugging
How we diagnosed the eval regression, rate limiting, and confidence tuning
Q
Your eval metrics dropped after the LangGraph migration. What happened?
▼

Average context_relevancy dropped from 0.833 → 0.518, answer_relevancy from 0.700 → 0.375. Two query types scored all zeros: ecosystem (4 queries) and auto (3 queries). Two queries outright failed.

Root cause: Groq rate limiting (HTTP 429). The LangGraph pipeline makes more LLM calls per query (classify + expand + generate + validate = 4), and the eval runner fires 30 queries with no delay. Total: ~210 LLM calls against a 30 RPM limit.

Rate limiting caused two cascade failures:

  1. Graph node 429: When classify/expand/generate hit 429, the entire graph.invoke() errored → query logged as "failed."
  2. Evaluator judge 429: The evaluator's _judge() method catches exceptions and returns 0.0. So even queries that produced answers got scored 0.0 because the judge itself was rate-limited.
Why ecosystem/auto hit hardest: They run last in the eval dataset (queries 22-30). By that point, the rate limit was fully saturated. Earlier queries (similar, swot) ran when there was still headroom.
Q
How did you fix the rate limiting issue?
▼

Two fixes:

  1. LLM_MAX_RETRIES = 3 — Added to settings.py, passed to OpenAI(max_retries=3). The OpenAI SDK has built-in exponential backoff. On 429, it waits 1s → 2s → 4s before giving up. Both the graph nodes AND the evaluator judge benefit (they both use get_llm_client()).
  2. QUERY_DELAY = 2 seconds between eval queries in run_eval.py. Spreads 210 LLM calls over ~60+ seconds instead of a tight loop. Keeps us under 30 RPM.
Q
How do you know the regression wasn't a real architecture problem?
▼

Diagnostic evidence:

  1. Live test succeeded: Running a single ecosystem query ("Startup ecosystem for mental health") produced best_score=2.862, confidence=high, correctly routed to generate_ecosystem. The graph logic was correct.
  2. 429 in the log: The live test's generate call hit HTTP/1.1 429 Too Many Requests. The LLM was reachable but rate-limited.
  3. Order-dependent failures: Ecosystem/auto failed because they run last. If it were an architecture bug, they'd fail every time regardless of order.
  4. Earlier types scored normally: similar (0.8 context_relevancy), competitor (0.92) — these ran first before rate limit kicked in.
Interview lesson: When eval metrics drop, don't assume the model is worse. Investigate infrastructure first — rate limits, API errors, network issues. The eval runner's infrastructure is as important as the pipeline's correctness.
Q
What's the "auto" type in the eval dataset? The graph never produces "auto".
▼

The eval dataset has 4 queries typed as "auto" — these are queries without an obvious type, like "Tell me about robotics startups" or "What's happening in the space tech industry?" In v3, the user could select "auto" from the dropdown, and the chain would use the auto prompt template.

In LG v1, the classify node maps these to a real type (usually "similar" or "ecosystem"). The eval dataset still labels them "auto" for the analysis_type column, but the graph internally classifies them. This is fine for eval — the evaluator uses the dataset's type label for structural checks but the actual answer is generated with the LLM-chosen type.

Q
What are your confidence thresholds and how did you pick them?
▼

Two thresholds from the cross-encoder reranker score:

ThresholdValueMeaning
CONFIDENCE_THRESHOLD_HARD0.02Below this → refuse. No LLM call.
CONFIDENCE_THRESHOLD_SOFT0.10Below this → low confidence. Generate with disclaimer.

These were calibrated empirically in v3 by testing with out-of-domain queries ("best pizza restaurants") and borderline queries. The cross-encoder returns scores roughly in [-10, +10] range, with relevant docs typically scoring > 1.0. The thresholds are deliberately very low — we'd rather generate with a disclaimer than refuse prematurely.

Deeper dives an interviewer might pursue
Q
How would you add a human-in-the-loop (HITL) approval step?
▼

LangGraph has built-in support for HITL via interrupt_before or interrupt_after:

# Add interrupt before validate: compiled = graph.compile( checkpointer=MemorySaver(), interrupt_before=["validate"], ) # First invoke pauses at validate result = graph.invoke({"user_query": "..."}) # → result contains answer draft for human review # Human reviews, then resumes: graph.invoke(None, config={"thread_id": "..."}) # → continues from validate → END

The architecture is already HITL-ready because we pass checkpointer=None in build_graph(). Swapping it for a MemorySaver or SqliteSaver enables persistence and resume.

Q
How would you add streaming to the generate nodes?
▼

LangGraph supports graph.stream() and graph.astream(). For token-level streaming from the LLM:

  1. Change graph.invoke() → graph.stream() in the Streamlit app
  2. Use the stream_mode="messages" option to get LLM tokens as they arrive
  3. In Streamlit, use st.write_stream() to render tokens progressively

The node code itself doesn't need to change — LangGraph handles the streaming plumbing at the framework level.

Q
If you could do the migration again, what would you do differently?
▼
  1. Start with Option B: We initially created 5 retrieve nodes (Option A) then flipped to 5 generate nodes. Should have recognized the asymmetry sooner — "branch where behaviour diverges."
  2. Smoke test after each node: Instead of writing all 12 nodes then testing, we should have tested classify alone, then classify+expand, building up incrementally. The Pydantic access bug would've been caught at node 1, not node 12.
  3. Rate limit awareness from day one: We added max_retries and eval delays only after seeing the regression. Should have budgeted LLM calls per query upfront (4-7 calls × 30 queries = 120-210 calls) and designed the eval accordingly.
  4. TypedDict first, Pydantic later: Start with TypedDict (simpler, dict access works) to validate the graph topology, then upgrade to Pydantic for production type safety. Reduces moving parts during the initial build.
Retrospectives are interview gold. Showing what you'd do differently demonstrates growth, self-awareness, and engineering maturity — more impressive than "it worked perfectly the first time."
Q
How do you test a LangGraph pipeline?
▼

Three levels:

  1. Unit test per node: Each node is a pure fn(state) → dict. Create a mock state, call the function, assert the returned dict. For factory nodes, pass a mock LLM client:
    # Test classify node mock_llm = MockLLM(response="swot") classify = make_classify_node(mock_llm) result = classify(MockState(user_query="SWOT of fintech")) assert result["analysis_type"] == "swot"
  2. Integration test per path: Test the full graph with a real (or mocked) retriever. Verify that "Find startups similar to Stripe" → classify=similar → expand → retrieve → confidence=high → generate_similar → validate → answer.
  3. Eval pipeline: 30-query benchmark with LLM-as-judge scoring (context_relevancy, groundedness, answer_relevancy). Automated regression detection.

Q
How does the graph handle errors mid-pipeline? What if the LLM call in expand_query fails?
▼

Each node handles errors at its own level:

  • expand_query: Catches exceptions and falls back to user_query as-is. Retrieval still works, just with a less optimal query.
  • classify: Catches exceptions and falls back to "similar" (most common type).
  • generate_*: No try/catch — if the LLM fails here, it throws. This is intentional: a failed generation means no answer. The run_eval.py catches this at the graph.invoke() level and logs it as a failed query.
  • retrieve: If retrieval fails, empty source_docs → confidence_gate sees no docs → confidence="none" → retry or refuse.

After adding max_retries=3 to the OpenAI client, transient 429/5xx errors are automatically retried with exponential backoff before reaching the node's error handling.

Q
How does the Mermaid visualization help? When do you use it?
▼

LangGraph auto-generates a Mermaid diagram from the compiled graph:

mermaid_code = graph.get_graph().draw_mermaid() # Returns a Mermaid string that renders as a flow chart

Key benefit: the diagram is generated from the actual code, not a manually drawn picture. If you add a node or change an edge, the diagram updates automatically. We use it in:

  • Notebooks: Visual verification that the graph topology matches our intent
  • README: The architecture diagram is a rendered Mermaid PNG
  • Debugging: When a query takes an unexpected path, the diagram helps trace which conditional edge fired

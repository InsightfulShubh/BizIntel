Why Guardrails? — The Problem v3 Solves
NEW in v3
❌ Without Guardrails
User: "Best pizza restaurants in NYC"
This is a STARTUP database — no pizza restaurants exist. But v1/v2 would retrieve the 5 "closest" garbage docs and ask the LLM to generate an analysis. The LLM hallucinates confidently.
Retrieved docs: 1. "FoodTech SaaS platform" (score: -2.1) 2. "Delivery logistics API" (score: -3.4) 3. "Restaurant POS system" (score: -4.0) LLM: "Here are the top pizza restaurants..." ← HALLUCINATED — none are restaurants!
→
✅ With Guardrails (v3)
Same query — now handled correctly
Reranker scores are checked. Best score -2.1 < 0.15 threshold → hard refusal. LLM is never called. User gets a helpful "I don't know" message.
Reranker scores checked: best_score = -2.1 (< 0.15 HARD threshold) → confidence = "none" → SKIP LLM (save API cost!) → Return: "🚫 I don't have enough information to answer this reliably."

5A
Cross-Encoder Reranker — Now Returns Scores
UPDATED in v3
rag/reranker.py
v2 Behaviour
rerank() → list[SearchResult]
Computed cross-encoder scores internally but discarded them — only returned sorted documents.
scored = sorted(zip(docs, scores), ...) reranked = [doc for doc, _score in scored] # ^^^^^^ discarded! return reranked # no scores
→
🆕 v3 Behaviour
rerank() → RerankedResults
RerankedResults(documents, scores)
New NamedTuple carries both documents AND their cross-encoder scores through the pipeline.
class RerankedResults(NamedTuple): documents: list[SearchResult] scores: list[float] # parallel, best-first top_scored = scored[:top_k] return RerankedResults( documents=[doc for doc, _ in top_scored], scores=[float(s) for _, s in top_scored], )
RerankedResults NamedTuple
Key insight: The cross-encoder was already doing the expensive work of scoring every (query, doc) pair. v3 simply preserves those scores instead of throwing them away. Zero extra compute cost.

rag/chain.py
🚫 Hard Refusal (_REFUSAL_MSG)
confidence = "none"
🚫 **I don't have enough information to answer this reliably.** The retrieved context doesn't appear relevant to your query. This can happen when: - The topic is outside the startup database's coverage - The query is too vague or ambiguous - No matching startups exist in our 134K dataset *Try rephrasing your query or broadening your search terms.*
⚠️ Soft Warning (_LOW_CONFIDENCE_DISCLAIMER)
confidence = "low"
⚠️ **Low confidence** — the retrieved context may not fully match your query. Results below should be reviewed critically. --- [normal LLM analysis follows here]

⚠️ Fallback when reranker is disabled: If RERANK_ENABLED = False, no cross-encoder scores are available. In that case, confidence defaults to "high" (optimistic) — the system behaves identically to v2. Guardrails require the reranker to function.

v3 Design Decisions — Why These Choices?
DecisionChoiceAlternatives ConsideredWhy This?
Confidence signal source Cross-encoder best_score Cosine distance, LLM self-rating, embedding distance Cross-encoder is the most accurate relevance signal — reads query+doc jointly. Already computed, zero extra cost.
Two-tier thresholds Hard (0.15) + Soft (0.4) Single threshold, three tiers, probabilistic Two tiers balance UX (most queries pass) with safety (garbage gets caught). Tuneable via config.
Hard refusal skips LLM Return immediately, no API call Call LLM anyway, let it decide If context is garbage, LLM can only hallucinate. Skipping saves cost + prevents misleading answers.
Include sources in refusal Yes — sources still returned Hide sources on refusal Transparency — user can inspect WHY the system refused. Builds trust.
No-reranker fallback Default to "high" (optimistic) Force reranker, use cosine as fallback Cosine distance is a weak proxy. Better to not guard than to guard poorly with wrong signal.
Score type NamedTuple (RerankedResults, RetrievalResult) Dict, dataclass, Pydantic model NamedTuple is lightweight, immutable, typed, and has tuple unpacking. Matches existing codebase style.

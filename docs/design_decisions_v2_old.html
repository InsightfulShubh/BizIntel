Cross-Encoder Reranker
Why reranking, what model, and how it plugs into the pipeline
Q
Why did we need a reranker when we already have semantic search?
▼
WHY

Semantic search uses a bi-encoder (all-MiniLM-L6-v2) — it encodes the query and each document independently into 384-dim vectors, then compares via cosine similarity. This is fast but approximate.

The problem: bi-encoders miss fine-grained relevance. Two documents about "payment technology" may have similar embeddings, but one might be about enterprise B2B payments (relevant to a Stripe query) while the other is about mobile wallets in Africa (less relevant).

Our v1 evaluation showed Context Relevancy = 0.70 — the weakest metric. 30% of retrieved docs were off-topic.

A reranker reads the (query, document) pair jointly through all transformer layers, catching nuances the bi-encoder misses.

Q
What is a cross-encoder and how does it differ from a bi-encoder?
▼
HOW
AspectBi-Encoder (retrieval)Cross-Encoder (reranking)
InputQuery and doc encoded separatelyQuery + doc concatenated → encoded together
OutputTwo vectors → cosine similaritySingle relevance score (logit)
Speed~1 ms per doc (precomputed)~7 ms per (query, doc) pair
AccuracyGood for recall (finding candidates)Much better for precision (ranking candidates)
ScalabilityScales to millions (HNSW index)Only practical on small candidate sets (10–100)

That's why we use them together: bi-encoder for recall (fetch 20 candidates), cross-encoder for precision (pick the best 5).

Q
Why did we choose cross-encoder/ms-marco-MiniLM-L-6-v2?
▼
ALTERNATIVES
ModelSizeSpeedQualityWhy Not?
ms-marco-MiniLM-L-6-v2 ✅22 MB~7 ms/pairGood— Chosen. Best speed/quality trade-off
ms-marco-MiniLM-L-12-v233 MB~14 ms/pairBetter2× slower, marginal accuracy gain
ms-marco-electra-base110 MB~30 ms/pairBestToo heavy for startup-style docs
Cohere Rerank APICloud~200 msExcellentPaid API, adds network latency
No reranker (v1)—0 msBaselineContext Relevancy stuck at 0.70

We chose the L-6 variant because: (1) 22 MB fits in memory easily, (2) ~150 ms total for 20 pairs is invisible to the user, (3) trained on MS MARCO passage ranking — matches our startup doc format.

Q
How does CrossEncoder.predict() work internally?
▼
CODE 
pairs = [(query, doc.text) for doc in candidates]  # 20 pairs
scores = model.predict(pairs)                       # → numpy array of 20 floats

# Internally:
# 1. Tokenise: "[CLS] query [SEP] document [SEP]"
# 2. Feed through 6 transformer layers
# 3. Take [CLS] output → linear layer → single logit
#
# score = 5.12  → very relevant
# score = -3.4  → irrelevant

# Sort descending, keep top 5
ranked_idx = np.argsort(scores)[::-1][:top_k]

Key insight: unlike bi-encoders, the cross-encoder sees both texts simultaneously, so it can do word-level attention across query↔document. That's why it's more accurate but can't be precomputed.

Q
Why over-fetch 20 docs instead of directly retrieving 5?
▼
WHY

The bi-encoder's ranking isn't perfect. Doc #7 (by bi-encoder) might actually be the most relevant one (by cross-encoder). If we only fetch 5, we'd never see it.

By over-fetching 20, the cross-encoder gets a broader candidate pool and can "promote" docs that the bi-encoder ranked too low.

Why 20 specifically? Empirical sweet spot — 10 is too few (misses good candidates), 50 adds ~350 ms latency for marginal gain. Research shows most of the gain comes from reranking 2–4× the final top-K.

Setting: RERANK_TOP_K_INITIAL = 20 in settings.py

Hybrid Search (Semantic + Keyword)
Why semantic-only search wasn't enough and how we combined two signal types
Q
What's the problem with semantic-only search?
▼
WHY

Semantic search (embeddings + cosine) excels at meaning but fails at exact keywords:

  • Company names: "Stripe" → the embedding captures "payment infrastructure" but might rank "PayPal" higher because it's semantically closer than a startup called "Stripes Inc."
  • Technical terms: "GraphQL API" → semantic search returns any API company, not specifically GraphQL ones
  • Acronyms: "NLP", "SaaS", "B2B" — embeddings may not distinguish these well

BM25 keyword search catches these exact matches because it's based on term frequency, not meaning.

Q
What is hybrid search and why is it better than either approach alone?
▼
HOW

Hybrid search combines two orthogonal retrieval signals:

SignalStrengthsWeaknesses
Semantic (embeddings)Synonyms, paraphrasing, conceptual similarityMisses exact keywords, company names, acronyms
Keyword (BM25)Exact term matching, rare words, namesNo understanding of meaning or synonyms

By fusing both, a doc that's both semantically similar AND contains the exact keywords gets boosted to the top. Research consistently shows hybrid search outperforms either signal alone.

Q
What alternatives to hybrid search did we consider?
▼
ALTERNATIVES
ApproachProsConsWhy Not?
BM25 + Semantic + RRF ✅Simple, effective, no trainingNeeds tuning weights— Chosen
SPLADE (learned sparse)Better than BM25Needs GPU, complex setupOverkill for our dataset
ColBERT (late interaction)Best qualityRequires pre-built index, heavyToo heavy for 134K docs
Dense-only + better embeddingsSimpleStill misses keyword matchesDoesn't solve the core problem
Query-time BM25 filterEasy to implementHard to tune thresholdLoses the ranking signal

BM25 + RRF is the industry standard used by Pinecone, Weaviate, and Elasticsearch. It's the simplest approach that gives the best quality/complexity trade-off.

Q
What is the complete retrieval pipeline in v2?
▼
CODE 
# retriever.py — 4-step pipeline

# Step 1: Semantic search (ChromaDB, HNSW index)
query_vec = embedder.encode_single(query)
semantic_results = store.query(query_vec, top_k=20)  # over-fetch

# Step 2: BM25 keyword search (in-memory index)
bm25_results = bm25_index.search(query, top_k=20)

# Step 3: Weighted RRF fusion (merge + deduplicate)
candidates = reciprocal_rank_fusion(
    ranked_lists=[semantic_results, bm25_results],
    weights=[1.0, 0.4],  # semantic dominates
    top_k=20,
)

# Step 4: Cross-encoder reranker (precision filter)
results = reranker.rerank(query, candidates, top_k=5)

Each step is toggleable via settings: HYBRID_SEARCH_ENABLED and RERANK_ENABLED.

Weighted Reciprocal Rank Fusion (RRF)
How we merge semantic and keyword results, and why we weight them unequally
Q
What is Reciprocal Rank Fusion (RRF)?
▼
HOW

RRF is a rank aggregation algorithm from SIGIR 2009 (Cormack, Clarke, Büttcher). It merges multiple ranked lists without needing scores to be comparable:

RRF_score(doc) = Σ  1 / (k + rank_in_list_i)

# k = 60 (standard constant that dampens rank influence)
# rank = 1-based position in each list

# Example:
# Doc_A is rank 1 in semantic, rank 3 in BM25:
#   score = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
#
# Doc_B is rank 2 in semantic only:
#   score = 1/(60+2) = 0.0161
#
# Doc_A wins because it appears in BOTH lists

Key insight: Documents found by both systems get boosted — this is the "agreement signal" that makes hybrid search powerful.

Q
Why did standard (unweighted) RRF cause a quality drop?
▼
PERFORMANCE

When we first added BM25 with equal weights (both 1.0), context relevancy dropped from 0.83 → 0.78.

Root cause: BM25 matches keywords literally. A query about "AI healthcare startups" would match docs containing "AI" or "healthcare" even if they're about "AI-powered gaming" or "healthcare furniture". These noisy keyword matches got equal voting power as semantic matches.

With equal weights, BM25's noisy docs could push down genuinely relevant semantic results.

Q
How did weighted RRF solve the problem?
▼
HOW 
RRF_score(doc) = Σ  weight_i / (k + rank_i)

# Weights:
#   Semantic: 1.0  (full influence — trusted signal)
#   BM25:     0.4  (reduced influence — useful but noisy)

# Now a doc ranked #1 by BM25 alone:
#   score = 0.4/(60+1) = 0.0066  (weak)
#
# A doc ranked #3 by semantic alone:
#   score = 1.0/(60+3) = 0.0159  (still stronger!)
#
# A doc in BOTH lists still gets the biggest boost

This lets semantic search dominate the final ranking while BM25 acts as a signal booster for docs that also have keyword matches. Best of both worlds.

Q
Why weights 1.0 and 0.4 specifically? How were they chosen?
▼
WHY

Chosen empirically based on our eval results:

ConfigurationContext RelevancyNotes
Semantic only (v1)0.70Baseline
Semantic + Reranker0.83Big improvement
Hybrid (1.0 / 1.0) + Reranker0.78BM25 noise hurt quality
Hybrid (1.0 / 0.4) + Reranker0.83Sweet spot — keyword boost without noise

The 0.4 weight means BM25 contributes ~29% of a semantic match's influence. Enough to boost relevant keyword matches, not enough to inject noise.

Settings: RRF_WEIGHT_SEMANTIC = 1.0, RRF_WEIGHT_BM25 = 0.4

Q
Why k=60 in the RRF formula?
▼
ALTERNATIVES

k=60 is the standard value from the original RRF paper. It controls how much rank position matters:

  • Small k (e.g. 1): Rank 1 gets a huge score (0.5), rank 2 gets much less (0.33). Very sensitive to rank position.
  • Large k (e.g. 60): Rank 1 gets 0.0164, rank 2 gets 0.0161 — small difference. More democratic, emphasises list agreement over exact rank.

k=60 is used by Elasticsearch, Pinecone, and most production systems. We didn't change it because it's well-validated and our candidate lists are small (20 each).

Q

▼

BM25 Keyword Search
Why BM25Okapi, how the index is built, and alternatives
Q
What is BM25 and how does it differ from TF-IDF?
▼
HOW

Both are bag-of-words retrieval models, but BM25 adds two improvements:

FeatureTF-IDFBM25Okapi
Term frequencyLinear: tf grows foreverSaturating: tf / (k₁ + tf) — diminishing returns after ~3 occurrences
Document lengthNo normalisationLength normalisation: longer docs are penalised (parameter b)
IDFlog(N/df)log((N - df + 0.5) / (df + 0.5)) — better for rare terms

BM25's saturation prevents a doc that mentions "payment" 50 times from dominating over one that mentions it 3 times — exactly the right behaviour for our startup descriptions.

Q
Why did we choose rank-bm25 library over alternatives?
▼
ALTERNATIVES
LibrarySizeSpeedFeaturesWhy Not?
rank-bm25 ✅8 KB~3s build, ~30ms queryBM25Okapi, pure Python + NumPy— Chosen. Tiny, no deps, fast enough
Elasticsearch / OpenSearch~500 MBVery fastFull-text, faceting, etc.Way too heavy — needs a Java server
Whoosh~1 MBFastFull Python search engineOverkill — we only need BM25 scoring
PyTerrier~50 MBFastResearch IR toolkitHeavy dependency (Java bridge)
Custom TF-IDF0~30msBasicNo term saturation, no length norm

rank-bm25 is perfect for our use case: 8 KB package, zero external deps, and handles 134K docs in 3 seconds.

Q
How is the BM25 index built and stored?
▼
CODE 
# bm25_search.py — index construction

# 1. Fetch all docs from ChromaDB
doc_ids, texts, metadatas = store.get_all_documents(batch_size=5000)

# 2. Tokenise each document
_TOKEN_RE = re.compile(r"[a-z0-9]{2,}")
tokenised = [_TOKEN_RE.findall(text.lower()) for text in texts]

# 3. Build BM25Okapi index (precomputes IDF, avg doc length)
bm25 = BM25Okapi(tokenised)
# → IDF table: {"payment": 4.2, "startup": 0.3, ...}
# → avg_doc_len: 47.3 tokens

The index is in-memory only — rebuilt on each app start (~3s). This is fine because:

  • Our corpus doesn't change at runtime
  • 3s startup is acceptable for Streamlit
  • No need for disk persistence complexity
Q

▼
ALTERNATIVES

Our tokeniser: re.compile(r"[a-z0-9]{2,}") — lowercase + keep alphanumeric tokens ≥ 2 chars.

TokeniserSpeed (134K docs)QualityWhy Not?

For BM25 on short startup descriptions, our regex is good enough. The IDF table handles most of the intelligence.

Q
How does get_all_documents() work and why was it needed?
▼
CODE

BM25 needs access to all document texts to build its IDF table. But our vector store interface (VectorStoreBase) only had query() for similarity search. We added:

# vectorstore/base.py — new abstract method
@abstractmethod
def get_all_documents(self, batch_size=5000):
    """Return (doc_ids, texts, metadatas) for ALL docs."""

# chroma_store.py — batched fetch (ChromaDB limit = 5000/call)
def get_all_documents(self, batch_size=5000):
    all_ids, all_texts, all_metas = [], [], []
    offset = 0
    while True:
        batch = self._collection.get(
            limit=batch_size, offset=offset,
            include=["documents", "metadatas"],
        )
        if not batch["ids"]:
            break
        all_ids.extend(batch["ids"])
        all_texts.extend(batch["documents"])
        all_metas.extend(batch["metadatas"])
        offset += batch_size
    return all_ids, all_texts, all_metas

Fetches all 134,092 docs in ~10s. One-time cost at startup.

Q
What is LLM-as-Judge and why use it instead of human evaluation?
▼
WHY

LLM-as-Judge = using a language model to score the output of another LLM. The "judge" LLM reads a structured prompt with rubric and outputs a numeric score.

Why not human evaluation?

  • We have 30 queries × 6 metrics = 180 judgements per eval run
  • Human eval would take hours; LLM-as-Judge takes minutes
  • Repeatable — same prompt always gives similar scores
  • Research shows GPT-4-class models correlate 0.85+ with human judges

Mitigation for judge bias:

  • Use temperature=0.0 for deterministic scoring
  • Structured rubric (0.0 / 0.5 / 1.0 anchors) in every prompt
  • 3 deterministic metrics as cross-check against LLM scores
Q
How does the _judge() helper work?
▼
CODE 
def _judge(self, prompt: str) -> float:
    response = self._client.chat.completions.create(
        model=self._model,        # groq or openai (via get_llm_client)
        messages=[{"role": "user", "content": prompt}],
        temperature=0.0,          # deterministic
        max_tokens=50,            # just need a number
    )
    text = response.choices[0].message.content.strip()

    # Extract first decimal number: "0.85" or "Score: 0.85"
    match = re.search(r"(\d+\.?\d*)", text)
    if match:
        return min(max(float(match.group(1)), 0.0), 1.0)  # clamp 0–1
    return 0.0  # fallback

Simple pattern: structured prompt → LLM → extract float. The regex handles various LLM output formats ("0.85", "Score: 0.85", "The relevancy is 0.85").

Q
Why 30 queries and not 10 or 100?
▼
WHY
  • 10 queries: Too few — won't cover all 6 analysis types and edge cases
  • 30 queries: 5 per analysis type + extra coverage — statistically meaningful averages, runs in ~10–15 minutes
  • 100 queries: Diminishing returns — each run costs ~300 LLM judge calls (3 per query) + 100 RAG pipeline calls. With Groq's 30 RPM limit, this would take 30+ minutes

30 is the sweet spot: large enough for statistical confidence, small enough to run frequently during development.

Q
How does the LLM client factory pattern work?
▼
CODE 
# config/llm_client.py

_API_KEY_ENV = {
    "openai": "OPENAI_API_KEY",
    "groq":   "GROQ_API_KEY",
}
_BASE_URLS = {
    "openai": None,                             # default OpenAI endpoint
    "groq":   "https://api.groq.com/openai/v1", # OpenAI-compatible
}

def get_llm_client(provider=None) -> OpenAI:
    provider = (provider or LLM_PROVIDER).lower()
    api_key = os.environ[_API_KEY_ENV[provider]]
    base_url = _BASE_URLS[provider]

    return OpenAI(api_key=api_key, base_url=base_url)

The key insight: Groq uses the OpenAI SDK with a different base_url. No new dependency needed. All 3 LLM call sites (query expansion, generation, eval judge) go through this one factory.

Q
Why a factory function instead of just changing the API key?
▼
WHY

With a factory, we get:

  • Single config change: Flip LLM_PROVIDER = "groq" → "openai" in settings.py. Every call site adapts.
  • Centralised validation: Missing API key? One error message, one place to fix.
  • Extensibility: Adding a new provider (e.g. Anthropic) = add 2 entries to the dicts + done.
  • Testability: Can override provider per-call: get_llm_client("openai")

Without the factory, we'd have if provider == "groq" scattered across 3 files — fragile and DRY-violating.

Q

▼

One setting controls provider + model + API key + endpoint — the rest is automatic.

Q
What is Groq's LPU and why is it so fast?
▼
HOW

LPU (Language Processing Unit) is Groq's custom ASIC chip designed specifically for sequential inference:

  • GPU approach: Massive parallelism, great for training, but LLM inference is sequential (token by token) — GPUs underutilise their hardware
  • LPU approach: Optimised for sequential processing, deterministic latency, no batching overhead

Result: Groq generates tokens at ~800 tokens/sec vs GPU-based APIs at ~50–100 tokens/sec. That's why our responses come back in ~0.5s instead of ~2s.

Pipeline Design & Architecture
Dependency injection, toggle flags, and how everything connects
Q
Why use dependency injection instead of hardcoding components?
▼
WHY 
# Without DI (hardcoded) — fragile:
class Retriever:
    def __init__(self):
        self.embedder = StartupEmbedder()       # can't swap
        self.store = ChromaStore()               # locked to Chroma
        self.reranker = StartupReranker()        # always loaded

# With DI (v2) — flexible:
class StartupRetriever:
    def __init__(self, embedder, store, reranker=None, bm25_index=None):
        self._embedder = embedder
        self._store = store
        self._reranker = reranker               # optional
        self._bm25 = bm25_index                 # optional

Benefits: (1) Swap ChromaDB ↔ FAISS without touching retriever, (2) disable reranker by passing None, (3) easy to test with mocks, (4) Streamlit caching works cleanly with injected singletons.

Q
Why use @st.cache_resource for loading models?
▼
HOW

Streamlit re-runs the entire script on every user interaction. Without caching, we'd reload:

  • Embedding model (~30s) on every chat message
  • Reranker model (~2s) on every chat message
  • BM25 index from 134K docs (~13s) on every chat message

@st.cache_resource loads each resource once, stores it in memory, and reuses across reruns:

@st.cache_resource
def load_reranker():
    return StartupReranker(RERANKER_MODEL_NAME)  # loaded once

@st.cache_resource
def load_bm25_index(_store):
    ids, texts, metas = _store.get_all_documents()
    return BM25Index(ids, texts, metas)          # built once
Q
What is the query expansion step and why is it important?
▼
WHY

Users type short, name-based queries like "Find startups similar to Stripe". The embedding model encodes this as-is, but it doesn't know what Stripe does.

Query expansion uses the LLM (temp=0.0) to rewrite the query into a rich description:

"Find startups similar to Stripe"
→ "online payment processing fintech developer API
   billing infrastructure internet commerce"

This richer text produces a much better embedding that matches relevant startup descriptions. Without expansion, we'd match "Stripe" the word — with expansion, we match the concept of payment infrastructure.

Q
What are the known limitations of v2?
▼
PERFORMANCE
  • Static corpus: No real-time data ingestion. Adding new startups requires re-embedding + reindexing BM25.
  • BM25 tokeniser is simplistic: No stemming, no stopword removal. "running" and "run" are different tokens.
  • No chunking: Each startup is one document. Longer descriptions might benefit from chunking.
  • Groq rate limits: 30 RPM means the eval run with 30 queries + 90 judge calls takes ~4 minutes.
  • No cross-lingual support: BM25 and embeddings are English-only.
  • Eval dataset not exhaustive: 30 queries covers common cases but may miss edge cases.
Q
What would you improve in a v3?
▼
ALTERNATIVES
  1. Better embeddings: Upgrade to all-mpnet-base-v2 (768-dim) or e5-large-v2 for higher recall
  2. Learned sparse retrieval (SPLADE): Replace BM25 with a learned sparse model that understands term importance
  3. Streaming responses: Stream LLM output to Streamlit for better UX
  4. Metadata pre-filtering: Filter by country/year before semantic search for speed
  5. A/B testing framework: Compare v1 vs v2 pipeline side-by-side on same queries
  6. Persistent BM25: Save/load BM25 index to disk to avoid rebuilding on each startup
  7. Multi-provider eval: Run eval with both Groq and OpenAI to compare judge quality
Q
What were the key lessons learned building v2?
▼
WHY
  1. Measure before optimising: The eval pipeline identified Context Relevancy as the bottleneck — without metrics, we'd have guessed wrong.
  2. Reranker > better embeddings: A 22MB cross-encoder gave +0.13 improvement — more than any embedding model upgrade would have.
  3. Hybrid search needs careful weighting: Equal weights made things worse. The "fix" was as simple as changing 1.0 to 0.4.
  4. OpenAI-compatible APIs are everywhere: Switching from OpenAI to Groq was a 2-line change (base_url + api_key) — no code rewrite.
  5. Toggle flags are essential: Being able to disable features via settings.py made debugging and A/B comparison trivial.
  6. DI pays off fast: Adding BM25 and reranker to the pipeline required zero changes to the chain, only to the retriever constructor.

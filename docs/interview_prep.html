01 High-Level Architecture
The big picture — why RAG, what pattern, how the layers connect
Q
What is BizIntel and what problem does it solve?
▼

BizIntel is an AI-powered Startup Intelligence Engine that answers natural-language questions about 134,000+ startups from Y Combinator and Crunchbase. It supports 6 analysis types: similarity search, SWOT, competitor analysis, side-by-side comparison, ecosystem mapping, and auto-detect.

Problem it solves: Traditional startup databases (Crunchbase, PitchBook) are search-by-filter tools. You can filter by industry, country, funding — but you can't ask "Find me startups solving the same problem as Stripe but focused on emerging markets". BizIntel bridges this gap using RAG (Retrieval-Augmented Generation) — combining semantic search with LLM reasoning.

Q
Why RAG and not just fine-tuning an LLM on your startup data?
▼
CriteriaFine-tuningRAG ✅
Data freshnessStale after training cutoff — must retrainUpdate CSV + re-embed; no model retraining
Hallucination controlLLM can generate fake companiesLLM is grounded — only sees retrieved docs
CostGPU hours for fine-tuning ($$$)Embedding once + cheap inference
TransparencyBlack box — can't show sourcesEvery answer shows the exact source documents
Data size134K records too large for context windowRetrieve top-K (5) relevant docs — fits easily
Key interview point: RAG gives us grounded, verifiable, up-to-date answers. Every claim in the response can be traced back to a real startup in our database. Fine-tuning would bake knowledge into model weights — making it impossible to verify or update cheaply.
Q
Walk me through the end-to-end architecture.
▼
OFFLINE PIPELINE (one-time batch job): CSV files → Data Cleaning → Document Builder → Embedder → Vector Store ↓ ↓ ↓ pandas + validation Pydantic SentenceTransformer → ChromaDB/FAISS ONLINE PIPELINE (per user query): User Query → Query Expansion (LLM) → Embedder → Vector Store → Top-K docs ↓ Prompt Template + Docs ↓ OpenAI LLM → Answer ↓ Streamlit UI (chat)

6 layers, each with a single responsibility:

  1. Config (settings.py) — Single source of truth for all parameters
  2. Preprocessing — Load, clean, validate, unify CSVs
  3. Embeddings — Document building + SentenceTransformer encoding
  4. Vector Store — Strategy Pattern (ChromaDB / FAISS)
  5. RAG — Retriever + Prompt Templates + Chain orchestrator
  6. App — Streamlit UI with state management
Q
Why a src/bizintel/ layout with pyproject.toml instead of flat scripts?
▼

We use the src-layout pattern recommended by the Python Packaging Authority (PyPA):

BizIntel/ ├── src/bizintel/ ← importable package │ ├── config/ │ ├── preprocessing/ │ ├── embeddings/ │ ├── vectorstore/ │ ├── rag/ │ └── app/ ├── scripts/ ← CLI entry points ├── pyproject.toml ← hatch build backend └── .python-version ← Python 3.13

Why not flat scripts?

  • Import isolation: src-layout prevents accidental import of local files vs installed package. Without it, import bizintel could resolve to the source dir OR the installed package — ambiguous.
  • Installable package: pip install -e . (or uv pip install -e .) makes bizintel importable from anywhere — scripts, notebooks, tests — without sys.path hacks.
  • Enterprise-grade: Shows interviewers you understand packaging, not just "it runs on my machine".
pyproject.toml uses hatchling as the build backend with [tool.hatch.build.targets.wheel] packages = ["src/bizintel"]. Hatch is the modern replacement for setuptools — faster, simpler, PEP 621 compliant.

02 Data Pipeline & Cleaning
How raw CSVs become a clean, unified dataset of 134K startups
Q
What are your data sources and why these two?
▼
SourceRowsStrengthWeakness
Y Combinator (2 CSV snapshots)~4,400High-quality curated startups, long descriptions, clean tagsSmall — only YC-funded companies
Crunchbase (1 CSV)~130KMassive coverage, global, includes non-VC companiesNoisy — many entities aren't real startups, short descriptions

Why both? YC gives us depth (rich descriptions, high signal). Crunchbase gives us breadth (global coverage). Together they cover the full spectrum. The source metadata field lets users filter by source in the UI.

Q
How do you unify two completely different CSV schemas?
▼

We define a STANDARD_COLUMNS schema in settings.py:

STANDARD_COLUMNS = [ "startup_id", "name", "description", "industry", "tags", "country", "founded_year", "source" ]

Each source has its own column rename map:

# YC: company_name → name, company_id → startup_id, year_founded → founded_year YC_COLUMN_MAP = {"company_id": "startup_id", "company_name": "name", ...} # Crunchbase: id → startup_id, country_code → country, founded_at → founded_at (parsed later) CRUNCHBASE_COLUMN_MAP = {"id": "startup_id", "country_code": "country", ...}

Process: Rename columns → apply source-specific cleaning → _finalize_dataframe() applies universal cleaning (text normalization, dedup, description length filter) → pd.concat([yc_df, cb_df]) merges into one unified DataFrame.

Why not join/merge? These aren't relational datasets with a foreign key. They're independent lists of companies. We concatenate (stack vertically), not merge (join horizontally). Dedup handles overlapping companies.
Q
How do you handle the description fallback chain?
▼

Crunchbase has 3 description fields with varying quality:

# Priority: description > overview > short_description df["description"] = ( df["description"] .replace("", pd.NA) .fillna(df["overview"]) .fillna(df["short_description"]) ) # If ALL are empty, build a synthetic description: fallback_text = "Startup in " + df["industry"] + " industry. Tags: " + df["tags"] df["description"] = df["description"].fillna(fallback_text)

Why a fallback chain? Without it, ~30% of Crunchbase records would have empty descriptions — making them invisible to semantic search. The synthetic fallback ensures every startup has some searchable text, even if minimal.

YC uses a simpler chain: long_description → short_description.

Q
How do you handle deduplication?
▼
df["name_lower"] = df["name"].str.lower() df = df.drop_duplicates(subset=["name_lower", "founded_year"]) df = df.drop(columns=["name_lower"])

Why name_lower + founded_year?

  • name_lower alone is too aggressive — "Apple" the fruit company vs "Apple" the tech company could exist.
  • Adding founded_year gives a second dimension. Two companies with the same name AND same founding year are almost certainly duplicates.
  • We don't use description for dedup because descriptions vary across sources even for the same company.
Known limitation: A company in both YC and Crunchbase with slightly different founding years (e.g., 2011 vs 2012 due to data entry) would NOT be deduplicated. A more robust approach would use fuzzy matching (e.g., Levenshtein distance + year tolerance), but for 134K records, exact match is a pragmatic starting point.
Q
What is the validation layer and why flag instead of delete?
▼

validation.py adds an is_suspicious boolean column based on 5 rules:

suspicious = ( (name_len < 2) # Name too short (e.g., "A") | (desc_len < 20) # Description too short | (name matches placeholder pattern) # "test", "example", "demo", "company1" | (founded_year < 1900) # Impossible year | (founded_year > current_year + 1) # Future year )

Why flag, not delete?

  • Non-destructive: Deleting records is irreversible. Flagging lets downstream consumers decide — the UI could filter them out, but a data analyst might want to inspect them.
  • Metadata-based filtering: is_suspicious is stored as metadata in the vector store, so you could add a where={"is_suspicious": "False"} filter at query time.
  • Auditability: You can always count flagged records to assess data quality.
Q
Why use ast.literal_eval for tags parsing?
▼

YC stores tags as Python list literals in CSV cells: "['B2B', 'SaaS', 'Fintech']". When pandas reads this, it's a string that looks like a list, not an actual list.

# Input: "['B2B', 'SaaS', 'Fintech']" ← string # After: "B2B, SaaS, Fintech" ← clean comma-separated string def _parse_tags(value): candidate = value.strip() if candidate.startswith("[") and candidate.endswith("]"): parsed = ast.literal_eval(candidate) # safely eval the list return ", ".join(str(v).strip() for v in parsed if str(v).strip()) return candidate

Why ast.literal_eval and not json.loads or eval?

  • eval() — DANGER executes arbitrary Python code. Never use on untrusted data.
  • json.loads() — Would fail because the list uses single quotes (Python syntax), not double quotes (JSON syntax).
  • ast.literal_eval() — SAFE only evaluates Python literals (strings, numbers, lists, dicts). Cannot execute arbitrary code.
03 Field Selection — Text vs Metadata
The most important architectural decision in any RAG system
Q
What goes into the embedded text vs metadata? Why?
▼

This is the most critical design decision in the entire pipeline. The split is defined in settings.py:

FieldText (Embedded)Metadata (Filtered)Why
name✅Semantically meaningful — "Stripe" maps to fintech in embedding space
industry✅Core semantic signal — "AI", "healthcare", "fintech" are what users search by
tags✅Rich keywords — "B2B, SaaS, API, payments" adds semantic dimensions
description✅The richest semantic content — what the company does
country✅✅ (DUAL)Embedded for "Indian startups" queries + metadata for exact filter {"country": "India"}
founded_year✅✅ (DUAL)Embedded for "recent startups" queries + metadata for range filter
startup_id✅Identifier only — no semantic meaning
source✅Not semantic — but needed for filtering (YC vs Crunchbase)
is_suspicious✅Quality flag — never embed quality signals
The DUAL_FIELDS concept is a key innovation: country and founded_year appear in BOTH text (for semantic similarity) AND metadata (for exact filtering). This gives users two ways to narrow results — semantic ("startups in India") and exact (sidebar filter for "India").
Q
Why put country and founded_year in BOTH text and metadata?
▼

Scenario 1 — Semantic query: User asks "AI startups in India founded after 2020". The word "India" and "2020" in the document text helps the embedding model find semantically relevant matches.

Scenario 2 — Exact filter: User selects "YC" in the sidebar → we pass where={"source": "YC"} to the vector store. If we also had a country dropdown, we'd pass {"country": "US"}.

If country was ONLY in text: We couldn't do exact filtering — the vector store would return "India" for queries about "Indiana" (semantic similarity).

If country was ONLY in metadata: Searching "Indian fintech" wouldn't match — the embedding wouldn't know the startup is in India.

DUAL = best of both worlds at the cost of slightly larger metadata storage (~8 bytes per field per doc).

Q
Why "Style C" (labeled key-value) for document text format?
▼

We considered 3 styles:

StyleExampleVerdict
A: Prose"Stripe is a fintech company founded in 2010 in the US…"❌ Hard to generate consistently from structured data; varies in quality
B: Raw concatenation"Stripe fintech B2B SaaS US 2010"❌ No structure — embedding model can't distinguish name from industry
C: Labeled key-value ✅"Name: Stripe\nIndustry: fintech\nTags: B2B, SaaS\nCountry: US\nFounded: 2010\nDescription: Online payments…"✅ Best of both — structured labels help the model weight each field properly
Name: Stripe Industry: fintech Tags: B2B, SaaS, payments, API, developer tools Country: US Founded: 2010 Description: Stripe is a technology company that builds economic infrastructure for the internet.

Why this works: The labels ("Name:", "Industry:") act as semantic anchors. When the embedding model sees "Industry: fintech", it understands that "fintech" is the industry — not just a random word. Research on dense retrieval (e.g., DPR, E5) shows that structured text with labels consistently outperforms raw concatenation for retrieval tasks.

Q
Why vectorized document building instead of row-by-row?
▼

We build all 134K documents using vectorized pandas/numpy — no Python for-loops over rows:

# Vectorized: np.where operates on entire columns at once labelled = np.where(cleaned != "", label + ": " + cleaned, "") # NOT: for i, row in df.iterrows(): ... (50x slower)

Performance comparison for 134K rows:

ApproachTimeWhy
df.iterrows()~60sPython loop overhead — each row is a separate Python operation
df.apply(lambda)~25sBetter but still row-by-row under the hood
np.where + pd.concat ✅~2sOperations happen in C/NumPy — no Python loop

Metadata building is also vectorized: df[meta_cols].to_dict("records") converts the entire DataFrame to a list of dicts in one C-level operation.

▼

04 Embedding Model & Strategy
Why all-MiniLM-L6-v2, batching, and 384 dimensions
Q
Why all-MiniLM-L6-v2 instead of OpenAI embeddings or a larger model?
▼
ModelDimensionSpeed (134K docs)CostQuality (STS Bench)
all-MiniLM-L6-v2 ✅384~20 min (CPU)$0 (local)82.4
all-mpnet-base-v2768~60 min (CPU)$0 (local)83.4
OpenAI text-embedding-3-small1536~15 min (API)~$8 for 134K docs~83
OpenAI text-embedding-3-large3072~20 min (API)~$16 for 134K docs~85

Why MiniLM?

  • Free & local: No API costs, no rate limits, no data leaving your machine — critical for 134K records.
  • Small & fast: 384-dim vectors are 4x smaller than OpenAI's 1536-dim, meaning 4x less memory in the vector store, 4x faster similarity computation.
  • Quality is sufficient: For structured labeled text (Style C), the 1-point quality gap vs larger models is negligible. Our labeled format compensates by giving the model more structure to work with.
  • Same model for index + query: Both document embeddings and query embeddings use the same model, ensuring they live in the same vector space.
When would you switch? If query quality degrades on complex queries (e.g., multi-concept like "B2B SaaS in healthcare with AI for emerging markets"), upgrade to all-mpnet-base-v2 (768-dim) or OpenAI embeddings. The Strategy Pattern in our vector store makes swapping trivial.
Q
Why manual batching instead of just model.encode(all_texts)?
▼
# Our approach: manual batching with logging for i in range(n_batches): batch_texts = texts[start_idx:end_idx] embeddings = self._model.encode(batch_texts, show_progress_bar=False) logger.info("Batch %d/%d done — %d docs in %.2fs", ...)

Three reasons:

  1. Observability: At 134K docs (~20 min), you NEED to know progress. SentenceTransformer's built-in progress bar works for notebooks but not for production logging. Our approach logs each batch with timing.
  2. Memory control: Encoding 134K texts at once requires holding all embeddings in memory simultaneously. Batching at 500 docs means only 500 × 384 × 4 bytes ≈ 0.75 MB per batch, not 134K × 384 × 4 bytes ≈ 200 MB all at once.
  3. Crash resilience: If batch 200/268 fails, you know exactly where to resume. With a single call, you'd lose everything.
Q
Why return numpy arrays instead of Python lists?
▼

StartupEmbedder.encode() returns np.ndarray of shape (N, 384):

  • Memory: NumPy stores 134K × 384 floats as a contiguous C array (~200 MB). A Python list of lists would be ~800 MB (each float is a 28-byte Python object).
  • Speed: FAISS index.add() and ChromaDB both accept numpy arrays natively. Converting from lists adds unnecessary overhead.
  • Slicing: embeddings[start:end] is a zero-copy view on numpy. On a list, it creates a new list.
  • Normalization: FAISS's normalize_L2() operates in-place on numpy — impossible with Python lists.
05 Vector Store — Strategy Pattern
ChromaDB vs FAISS, cosine similarity, factory function
Q
Why the Strategy Pattern (ABC + Factory) for vector stores?
▼
# ABC base class class VectorStoreBase(ABC): @abstractmethod def count(self): ... @abstractmethod def add(self, texts, embeddings, metadatas): ... @abstractmethod def query(self, query_embedding, top_k, where): ... @abstractmethod def reset(self): ... # Factory function def create_vector_store(backend="chroma"): if backend == "chroma": return ChromaStore() if backend == "faiss": return FAISSStore() # Usage — the rest of the code doesn't know which backend it's using store = create_vector_store(settings.VECTOR_STORE_BACKEND)

Why this pattern?

  1. Open/Closed Principle: Adding a new backend (e.g., Pinecone, Weaviate, Qdrant) means creating a new class + one line in the factory. Zero changes to Retriever, Chain, UI.
  2. Testability: Unit tests can inject a mock vector store. Integration tests can use FAISS (fast, no server) while production uses ChromaDB.
  3. Config-driven swap: Change VECTOR_STORE_BACKEND = "faiss" in settings.py — entire system switches backends without touching any other file.
  4. Lazy imports: The factory uses from ... import ChromaStore inside the if-block, so FAISS is never imported if using ChromaDB (and vice versa). This avoids import errors if one library isn't installed.
Q
ChromaDB vs FAISS — when to choose which?
▼
FeatureChromaDB ✅ (Our default)FAISS
Setup complexityZero — stores everything in one dirNeed JSON sidecar for text + metadata
Metadata filtering✅ Native — where={"source": "YC"}❌ Post-filter only (over-fetch then discard)
PersistenceBuilt-in PersistentClientManual — faiss.write_index() + JSON save
ScalabilityGood to ~1M docsExcellent — handles billions with IVF/PQ indexes
GPU support❌ Not direct✅ faiss-gpu for massive scale
Approximate NNHNSW (built-in)IVF, PQ, HNSW — configurable
EcosystemPython-first, great for prototypingFacebook Research, industry standard

Why ChromaDB as default: For 134K docs, ChromaDB's native metadata filtering and zero-config persistence are worth more than FAISS's raw speed. FAISS becomes essential at 10M+ docs or when GPU acceleration is needed.

FAISS trick: We use IndexFlatIP (inner product) with normalize_L2() on all vectors. Inner product on L2-normalized vectors = cosine similarity. This is mathematically equivalent to IndexFlatL2 with a distance-to-similarity conversion, but simpler — higher score = more similar.
Q
Why cosine similarity and not Euclidean distance or dot product?
▼
MetricWhat it measuresIssue for text
Euclidean (L2)Straight-line distance in vector spaceSensitive to magnitude — a long document about fintech might be farther from a short fintech query than a long healthcare document
Dot productProjection of one vector onto anotherBiased toward longer vectors — favors documents with more text
Cosine similarity ✅Angle between vectors (direction, not magnitude)✅ Ignores magnitude — a 3-word fintech description and a 100-word one are equally close to a fintech query if they point the same direction

In our system: Document lengths vary wildly — YC descriptions average ~200 chars, Crunchbase ~50 chars. Cosine similarity ensures short descriptions aren't disadvantaged.

Implementation:

  • ChromaDB: metadata={"hnsw:space": "cosine"} on collection creation
  • FAISS: faiss.normalize_L2(vectors) + IndexFlatIP — mathematically equivalent to cosine
Q
How does FAISS handle metadata filtering without native support?
▼

FAISS only stores vectors — no text, no metadata. Our FAISS backend uses a JSON sidecar pattern:

# On add: store text + metadata in a JSON file self._docstore[f"doc_{j}"] = {"text": texts[j], "metadata": metadatas[j]} self._save_docstore() # writes to docstore.json # On query: over-fetch then post-filter fetch_k = top_k * 3 if where else top_k # fetch 3x more distances, indices = self._index.search(query_vec, fetch_k) for dist, idx in zip(distances, indices): entry = self._docstore[f"doc_{idx}"] if where and not self._matches_filter(entry["metadata"], where): continue # skip non-matching results.append(...) if len(results) >= top_k: break

The 3x over-fetch strategy: If user wants top-5 with source=YC, we fetch top-15 from FAISS (without filter), then discard Crunchbase results in Python. The 3x multiplier is a heuristic — works well when the filtered subset is ≥30% of the data.

Trade-off: Post-filtering can return fewer than top_k results if the filter is very selective (e.g., filtering for a rare country). ChromaDB's native filtering doesn't have this issue.
06 RAG Chain & Query Expansion
The core intelligence — how queries become grounded answers
Q
What is query expansion and why was it critical?
▼

The problem we discovered: When a user asks "Find startups similar to Stripe", the embedding model encodes the word "Stripe" as-is. But our startup documents don't contain the word "Stripe" — they describe their OWN products. The embedding for "Stripe" is close to documents containing the word "Startup" (string similarity) rather than documents about fintech payments.

Before query expansion:

Query: "Find startups similar to Stripe" Results: StartupBus, Startup Genome, Startupbootcamp ← matched on "Startup", not fintech!

After query expansion:

Query: "Find startups similar to Stripe" ↓ LLM expands to: "online payment processing fintech developer API billing infrastructure internet commerce" ↓ Embedding matches on actual business domains Results: Paid, Allied Payment Network, Intrannuity ← real fintech companies!

How it works: Before embedding the query, we send it to the LLM with a specialized prompt that rewrites it into domain vocabulary:

_EXPAND_PROMPT = """You are a search query optimizer for a startup database. Rewrite it into a rich, descriptive search query that captures the BUSINESS DOMAIN, INDUSTRY, KEY PRODUCTS, and TECHNOLOGY. Output ONLY the rewritten query, nothing else. Examples: User: Find startups similar to Stripe Rewritten: online payment processing fintech developer API billing infrastructure User: competitors of Airbnb Rewritten: short-term rental marketplace vacation home booking hospitality travel"""
This is a form of "HyDE" (Hypothetical Document Embeddings) — a technique where you use the LLM to generate a hypothetical document that would answer the query, then embed THAT instead of the raw query. Our approach is a lightweight version: instead of generating a full hypothetical document, we generate expanded search terms.
Q
Why Dependency Injection in the Retriever?
▼
class StartupRetriever: def __init__(self, embedder: StartupEmbedder, store: VectorStoreBase): self._embedder = embedder # injected, not created self._store = store # injected, not created # The retriever doesn't know or care which vector store it's using retriever = StartupRetriever(embedder, chroma_store) # or faiss_store

Benefits:

  • Testability: In tests, inject a mock embedder that returns fixed vectors + a mock store that returns fixed results. Test retrieval logic without loading a 90MB model.
  • Decoupling: Retriever doesn't import ChromaStore or FAISSStore. It works with any VectorStoreBase. Swap backends without touching retriever code.
  • Single Responsibility: Retriever's only job: encode query + call store. It doesn't manage embedder lifecycle, store connections, or config.
Q
Why does the Chain use the original query for the prompt but the expanded query for retrieval?
▼
# Step 1: Expand for RETRIEVAL (semantic search) search_query = self._expand_query(query) # "Stripe" → "fintech payments API..." results = self._retriever.retrieve(query=search_query, ...) # Step 2: Use ORIGINAL query for LLM PROMPT prompt = get_prompt(analysis_type, query=query, documents=...) # ↑ original "Find startups similar to Stripe"

Why? The expanded query is optimized for the embedding model — it's a bag of keywords. But the LLM needs the user's original intent to generate a coherent response. If we sent "online payment processing fintech developer API billing" to the LLM, it wouldn't know the user was asking about Stripe specifically.

Q
Why TOP_K = 5 as the default?
▼
  • Context window budget: Each document is ~200-500 chars of Style C text. 5 docs ≈ 1,500–2,500 chars ≈ 500-800 tokens. GPT-4o-mini has 128K context, but more context = more cost + slower + more noise.
  • Relevance decay: In practice, results #6+ often have low relevance scores. Including them adds noise to the LLM prompt.
  • UX: Users can cognitively process ~5 source cards. 20 source documents would overwhelm the UI.
  • Configurable: The sidebar slider lets users increase to 20 for broader exploration or decrease to 3 for focused analysis.

GPT-4o-mini, temperature tuning, 6 specialized templates
Q

▼

Key reasoning: Our prompts are well-structured with clear formatting instructions. GPT-4o-mini follows structured prompts extremely well — the quality gap vs GPT-4o is negligible for our use case. At 17x lower cost, it's the pragmatic choice.

Q
Why temperature=0.3 and not 0?
▼
  • Temperature 0: Deterministic — always the same output for the same input. Good for factual Q&A, but produces robotic, repetitive text for analysis.
  • Temperature 0.3: Mostly deterministic with slight creativity. The LLM can rephrase, find interesting angles, vary sentence structure — while staying grounded in the data.
  • Temperature 0.7-1.0: Too creative for data analysis — might hallucinate connections or make unsupported claims.

Exception: Query expansion uses temperature=0.0 because we want consistent, predictable keyword expansion — not creative rewriting.

Q
Why 6 separate prompt templates instead of one generic prompt?
▼

One generic prompt ("Analyze these startups based on the user's question") produces inconsistent output — sometimes a list, sometimes prose, sometimes a table.

6 specialized templates give the LLM explicit output structure:

TypeTemplate ForcesExample Output Structure
autoLLM decides best formatVaries — good for ambiguous queries
similarNumbered list with similarity explanations"1. Paid (US, 2019) — Similar because..."
swot## Strengths / Weaknesses / Opportunities / ThreatsStructured 4-quadrant analysis
competitorDirect / Indirect / Differentiators / PositioningCompetitive landscape map
comparisonMarkdown table + narrativeSide-by-side comparison grid
ecosystemOverview / Key Players / Sub-segments / Trends / GapsIndustry ecosystem map

The "_BASE_ROLE" prefix is shared across all templates — it tells the LLM it's "BizIntel, an AI-powered startup intelligence analyst" with access to 134K+ startups and must use ONLY the provided data. This grounding instruction prevents hallucination.

Q
How do you prevent the LLM from hallucinating startup data?
▼

Four layers of defense:

  1. Grounding instruction in prompt: "Use ONLY the startup data provided below to answer. If the data is insufficient, say so clearly."
  2. Only retrieved documents in context: The LLM never sees the full database — only 5 relevant documents. It can't make up companies that aren't in the context.
  3. Source cards in UI: Every answer shows the exact documents used. Users can verify claims against sources.
  4. Low temperature (0.3): Reduces creative/hallucinatory tendencies.
Verified in testing: We queried "Find startups similar to Stripe" and confirmed every result (Paid, Allied Payment Network, Intrannuity) exists in our dataset with matching metadata (source=YC/Crunchbase, correct countries and founding years).

Q
Why @st.cache_resource and not @st.cache_data?
▼
@st.cache_resource(show_spinner="Loading embedding model…") def load_embedder() -> StartupEmbedder: return StartupEmbedder() # loads 90MB model @st.cache_resource(show_spinner="Connecting to vector store…") def load_vector_store() -> VectorStoreBase: return create_vector_store() # opens ChromaDB connection

Streamlit reruns the entire script on every user interaction (every click, every keystroke). Without caching, we'd reload the 90MB embedding model and reconnect to ChromaDB 50+ times per session.

DecoratorReturnsSerializationUse For
@st.cache_dataSerializable data (dicts, DataFrames)Pickles the return valueData transformations, CSV loading
@st.cache_resource ✅Shared objects (models, DB connections)Stores the reference (no copy)ML models, DB clients, API clients

Why not @st.cache_data? Our embedder contains a PyTorch model — it can't be pickled/serialized efficiently. @st.cache_resource stores the object reference directly, so all users share the same model instance in memory.

Q
How do you handle the "quick action buttons" triggering a query?
▼
# In render_welcome(): if st.button(icon_label, key=f"ex_{icon_label}"): st.session_state.messages.append({"role": "user", "content": query}) st.session_state._pending_query = query st.rerun() # force a re-execution # At the bottom of streamlit_app.py: pending = st.session_state.pop("_pending_query", None) if pending: _process_query(pending)

The trick: Streamlit button clicks trigger a rerun. We can't call _process_query() directly inside the button handler because the function definition might not exist yet (Streamlit's top-to-bottom execution model). So we:

  1. Store the query in st.session_state._pending_query
  2. Call st.rerun() to restart the script
  3. At the bottom of the script, check for pending queries and process them

Q
Why not chunk the documents? RAG systems typically use chunking.
▼

Chunking is for long documents (PDFs, articles, books) where a single document exceeds the embedding model's token limit (typically 256-512 tokens). Our startup documents are short:

  • Average document: ~100-250 tokens (Style C format)
  • Max document: ~400 tokens (rich YC descriptions)
  • Model limit: 512 tokens (all-MiniLM-L6-v2)

Every startup is ONE document = ONE embedding. No need to chunk because:

  1. Documents fit within the model's context window
  2. Each startup is an atomic entity — splitting it would lose meaning
  3. No chunk-boundary issues (e.g., a sentence split across two chunks)
  4. Retrieval is at startup level, not passage level — we want to find companies, not paragraphs
When would you chunk? If we added long-form data (press releases, SEC filings, blog posts), we'd need chunking. A good strategy: RecursiveCharacterTextSplitter at 500 tokens with 50-token overlap, preserving the startup_id in chunk metadata for parent-document retrieval.
Q
How does ChromaDB's HNSW index work at a high level?
▼

HNSW (Hierarchical Navigable Small World) is an approximate nearest-neighbor algorithm:

  1. Multi-layer graph: Imagine a skip-list but for vectors. Top layers have few nodes (long-range connections), bottom layers have all nodes (short-range connections).
  2. Search: Start at the top layer, greedily walk toward the query vector. Drop to lower layers for finer navigation. Like zooming in on Google Maps.
  3. Trade-off: ef_construction (build quality) and ef_search (search quality) control speed vs accuracy. Higher = more accurate but slower.

For our 134K dataset: HNSW query takes ~5ms (vs ~50ms for brute-force flat scan). At 134K documents, even brute force is fast enough, but HNSW gives us headroom to scale to millions.

Contrast with FAISS IndexFlatIP: Our FAISS backend uses exact (brute-force) search — it checks every vector. This is simpler, always correct, but O(n). HNSW is O(log n) amortized. For 134K, the difference is negligible; for 10M+, HNSW wins decisively.

# Our approach — direct SDK call response = self._client.chat.completions.create( model=self._model, messages=[{"role": "user", "content": prompt}], temperature=self._temperature, max_tokens=self._max_tokens, ) answer = response.choices[0].message.content

Why direct?

  • Debugging: If something goes wrong, the stack trace points to our code, not into a framework's internals.
  • Minimal dependency: openai SDK is the thinnest possible layer over the HTTP API. Frameworks add layers that can break, have bugs, or change APIs.
  • Easy to swap: To switch to Claude, replace OpenAI() with Anthropic() and adjust the message format. The interface is nearly identical.

Q
Walk me through exactly what happens when a user types "Find startups similar to Stripe".
▼
  1. Streamlit captures input → st.chat_input() returns "Find startups similar to Stripe"
  2. Message saved → appended to st.session_state.messages
  3. chain.analyze() called with query="Find startups similar to Stripe", analysis_type="similar"
  4. Query expansion → LLM call (temp=0.0): "Stripe" → "online payment processing fintech developer API billing infrastructure internet commerce"
  5. Retriever.retrieve() → embedder.encode_single(expanded_query) → 384-dim numpy vector
  6. Vector store query → ChromaDB .query(embedding, top_k=5) → returns 5 SearchResults with text + metadata + cosine distances
  7. Prompt building → get_prompt("similar", original_query, documents_text) → complete prompt with retrieved docs + formatting instructions
  8. LLM call → OpenAI chat.completions.create() with the prompt → gets structured analysis
  9. Response packaging → {"answer": "...", "sources": [...], "analysis_type": "similar", "model": "gpt-4o-mini"}
  10. UI rendering → st.markdown(answer) + render_sources(sources) + timing footer
  11. State saved → response appended to st.session_state.messages for chat history

Total latency breakdown:

Query expansion: ~0.5s (LLM call, temp=0.0, 120 tokens) Query embedding: ~0.01s (single text, local model) Vector search: ~0.01s (ChromaDB HNSW on 134K docs) Prompt building: ~0.001s (string formatting) LLM generation: ~2-4s (gpt-4o-mini, ~500 output tokens) ──────────────────────────── Total: ~3-5s

Q
What was the hardest bug you encountered and how did you fix it?
▼

The semantic mismatch bug.

When we first tested "Find startups similar to Stripe", the results were StartupBus, Startup Genome, Startupbootcamp — companies with "Startup" in their name, not fintech companies.

Root cause analysis:

  1. The embedding model encodes "Stripe" as a short, ambiguous token
  2. The word "Startup" appears in many company names, creating a high-frequency cluster in embedding space
  3. Cosine similarity between "Stripe" and "StartupBus" was higher than between "Stripe" and "online payment processing" because the embedding model doesn't have world knowledge about what Stripe does

Solution: Query expansion — use the LLM (which DOES have world knowledge) to translate "Stripe" into the domain vocabulary that actually appears in our documents: "online payment processing fintech developer API billing infrastructure".

Result: After adding _expand_query() (15 lines of code), the same query returned Paid, Allied Payment Network, Intrannuity — real fintech companies verified in our dataset.

Interview framing: "This bug taught me that embedding models are encoders, not knowledge bases. They encode text into geometric positions based on linguistic patterns. For proper noun queries, you need a bridge — an LLM that can translate proper nouns into the domain vocabulary that the embedding model was trained on."
Q
Why not use a knowledge graph instead of/alongside vector search?
▼

A knowledge graph (e.g., Neo4j) would model explicit relationships: "Stripe → funded_by → YC", "Stripe → competes_with → Adyen". This is powerful but:

  • We don't have relationship data. Our CSVs are flat records — no investor, acquisition, or partnership edges. Building a KG from scratch would require entity resolution and relationship extraction (a separate ML pipeline).
  • Semantic search is more flexible. A KG can answer "Who competes with Stripe?" only if that edge exists. Vector search answers it by finding companies with similar descriptions — even if no explicit "competes_with" edge was ever created.
  • Hybrid opportunity: In a future version, we could add a KG for explicit relationships (investor, market) and use vector search for implicit similarity. This is called GraphRAG — Microsoft Research has shown it significantly improves RAG quality for complex queries.
